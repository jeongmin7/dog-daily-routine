## 배운 개념 (백엔드)

1. **세션 스코프 = 보안의 핵심.** `where: { userId }` 한 줄이 "남의 데이터 차단"을 담당. 목록(`findMany`)뿐 아니라 단건도 `where: { id, userId }`로 id+소유를 동시에 검사.
2. **클라 입력 불신(2차 검증).** 프론트가 검사해도 백엔드가 다시 검사. `userId`는 절대 클라(body)에서 받지 않고 서버 세션이 정한다 → body로 받으면 남의 계정에 데이터 심기 가능.
3. **`findUnique` vs `findFirst`.** `findUnique`는 PK/unique 조합만 `where`에 받음 → 일반 필드(`userId`) 같이 걸려면 `findFirst`.
4. **`findMany`는 빈 결과도 `[]`** (null 아님) → 0마리여도 에러 없음.
5. **404 vs 403.** "내 것 아님"도 404로 통일 → 리소스 존재 여부 노출 방지.
6. **동적 라우트 시그니처(Next 16).** 핸들러 첫 인자는 항상 `Request`, `params`는 **두 번째** 인자이며 **`Promise`** → `const { id } = await params`. 옛 버전(객체 그대로)과 달라짐. (AGENTS.md "추측 말고 문서 봐라"의 실제 사례)

## 막혔던 것 / 디버깅

- **`prisma.user.dogs` 같은 경로는 없다.** Prisma 클라이언트는 최상위 모델(`prisma.dog`)만 노출. `dogs`는 User 레코드의 관계 필드일 뿐.
- **스키마에 없는 필드(`age`) 저장 시도** → 타입+런타임 에러. Dog는 `birthdate`/`weight`만 있음(나이는 생일로 저장하고 계산).
- **동적 라우트 시그니처 오작성** → Next가 `.next/dev/types/`에 자동 생성한 검증기가 `tsc`에서 잡아줌("첫 인자는 NextRequest인데 id가 없다"). → **타입 체커가 1차 리뷰어** 역할.
- 문법 에러(중괄호 누락)가 있으면 컴파일러가 거기서 멈춰 그 뒤(`age` 에러)를 못 봄 → 에러는 위에서부터 하나씩.
